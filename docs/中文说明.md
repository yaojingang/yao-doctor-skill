# 中文说明

`yao-doctor-skill` 是一个面向本地 skill 库和 AI 工作台配置面的安全审计 skill。它的核心目标不是简单地把“权限大”判成“危险”，而是把：

- `能力风险（capability risk）`
- `不安全行为（unsafe behavior）`

明确拆开，再结合证据强度给出最终处置建议。

## 它解决什么问题

在 skill、agent、CLI 工作台和自动化配置越来越多之后，真正的问题往往不是“能不能读文件”，而是：

- 会不会读取隐私或凭据
- 会不会静默外发
- 会不会下载后执行
- 会不会通过 workflow、rules、hooks、automation 这类高杠杆入口改变系统行为

`yao-doctor-skill` 用一套偏务实的静态审计方式，帮助你在大量 skill 里快速分清：

- 哪些只是能力面大，但暂时没有强恶意证据
- 哪些已经出现了比较明确的不安全信号
- 哪些需要优先人工复核

## 默认扫描范围

当前版本默认会自动发现并扫描这些目标：

- 当前工作区及其常见 skill 根目录
- `~/.agents/skills`
- `~/.openclaw/skills`
- `~/.codex/skills`
- `~/.codex/plugins/cache`
- `~/.claude/skills`
- `~/.codex` 和 `~/.claude` 的部分工作台配置面
- 项目级 `AGENTS.md`、`CLAUDE.md`、选定的 `.claude/.codex` 配置、rules、hooks、automation 入口

它同时支持：

- 普通 skill 包
- plugin cache 里的 skill
- workbench surface
- 项目级 agent surface

## 核心判断逻辑

### 1. 能力风险分

能力风险分描述的是这个目标“有能力做什么”，常见维度包括：

- 文件读取
- 命令执行
- 环境变量访问
- 网络外发
- 远程安装
- 持久化
- 自动化控制

能力面大不等于不安全，但说明它一旦被滥用，攻击面会更大。

### 2. 不安全行为分

不安全行为分关注的是“是否已经出现了更接近真实危险的行为证据”，例如：

- 隐私读取
- 凭据读取
- 敏感源到外发出口的链路
- 下载后执行
- base64 + exec
- 压缩包落地后再执行
- 短链或粘贴站下载
- 隐蔽或规避边界的行为

### 3. 最终处置

最终处置不是只看权限多少，而是综合能力面、行为证据、保护面和证据置信度后落成：

- `safe`
- `risky`
- `suspicious`
- `unsafe`
- `critical`

同时映射为更便于执行的动作建议：

- `observe`
- `review`
- `block`
- `quarantine`

## 报告怎么读

生成的 HTML 报告当前固定为一套比较稳定的结构：

### 1. 报告概览

先看全库层面的总体分布，帮助你决定先审哪里。这里会展示：

- 处置结果分布
- 最高不安全分模块
- 能力风险和不安全行为的二维象限

### 2. 数据分析

把当前扫描到的目标按运行形态做四类拆分，而不是只按名字猜：

- 控制面型
- 外部集成型
- 本地操作型
- 指令轻量型

每一类都会展示：

- 数量和占比
- 风险构成
- 活跃率代理、平均能力风险分、平均不安全行为分
- 类型特点和使用信号说明

说明：当前“使用频率”并不是真实执行次数，而是基于近期活跃度的代理信号。

### 3. 指标定义

这里是整个报告的说明书，帮助你理解：

- 能力风险分代表什么
- 不安全行为分代表什么
- 各分段如何划分
- 最终处置梯度如何解释

### 4. 模块说明

这一部分告诉你每个 skill 模块默认是如何展开阅读的：

- 先看结论标签
- 再看三张图
- 最后回到证据

### 5. 技能模块

每个 skill 模块默认包含：

- 元信息卡
- 双分值对比
- 发现严重度分布
- 能力画像
- 审计意见
- 主要证据
- 能力证据
- 分析总结与建议说明

这些内容的目标不是替你最终拍板，而是让你更快定位“为什么命中、命中了什么、要不要继续看代码”。

## 什么时候适合用

比较适合这些场景：

- 扫描本地 skill 库
- 扫描 Codex、Claude、OpenClaw 等工作台入口
- 做批量 skill 安全体检
- 在提交到 GitHub 之前做一轮安全审阅
- 在团队内建立一套统一的 skill 审计报告

## 它不是什么

`yao-doctor-skill` 目前更像一个高质量的安全审计助手，而不是自动替你做最终裁决的“安全判官”。

它已经补了：

- 误报收敛
- obfuscation 规则
- 保护面提级
- 增量扫描
- 证据置信度分层
- 部分跨文件语义传播

但它仍然不是完整的动态沙箱，也不是全语义污点引擎。对于高风险目标，最终还是建议回到具体代码、具体用途和你自己的使用场景里做人工复核。

## 快速开始

在仓库根目录运行：

```bash
python3 scripts/run_yao_doctor_skill.py --full-scan
```

生成后会得到：

- `report.json`
- `report.md`
- `report.html`

同时会维护两个稳定入口：

- `_yao_doctor_skill_reports/full-library-latest/report.html`
- `_yao_doctor_skill_reports/changed-only-latest/report.html`

校验当前报告 UI 合约：

```bash
python3 scripts/validate_report_ui_contract.py _yao_doctor_skill_reports/full-library-latest/report.html
```

为了避免把真实本地扫描结果直接放到公开仓库里，仓库额外提供了一份完全虚构的示例报告，专门用于：

- 展示报告结构
- 演示 UI 和交互
- 作为对外说明材料

对应文件在：

- `docs/example-report/report.html`
- `docs/example-report/report.json`
- `docs/example-report/report.md`

这份示例里的 skill 名、路径、用途、风险和统计数字都不是来自真实本地环境。

## 说明与边界

- 本仓库默认把生成报告和缓存目录加入 `.gitignore`
- 报告页面逻辑、排版、UI 和交互当前已经固化成一套可校验契约
- 报告中的结论仅供建议和参考
- 实际情况仍然需要结合你的使用场景、真实用途和代码上下文做评估
- 对于 `unsafe` 和 `critical`，建议进一步查看对应代码和用途，再决定是否阻止或隔离
